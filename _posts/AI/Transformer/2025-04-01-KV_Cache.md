캐시(Cache)란 자주 사용되거나 다시 계산하는 데 비용이 큰 데이터를 미리 저장해 두었다가, 필요할 때 빠르게 재사용하기 위한 임시 저장소를 의미한다.

KV Cache에서의 Cache 역시 같은 개념이다.
<br>
트랜스포머 모델이 생성 과정에서 계산한 Key와 Value 벡터를 저장해 두고, 이후 토큰 생성 시 이를 재활용하는 기법을 의미한다.

## 기존 트랜스포머에서의 문제점

LLM은 텍스트를 생성할 때 Autoregressive 방식을 사용한다.
<br>
즉, 이전까지 생성된 토큰들을 바탕으로 다음 토큰을 하나씩 예측하며 문장을 완성해 나간다.

예를 들어 GPT에게 다음과 같은 질문을 하고 답변을 얻었다고 가정해 보자.

```
User: Can you explain LLMs?

GPT : Sure. Large Language Models (LLMs) are AI systems trained to understand 
      and generate human language.
```

이 답변은 실제로 한 번에 생성되는 것이 아니라, 아래와 같이 토큰 단위로 순차적으로 생성된다.

- **Step1:** $\mathbf{x}=\text{Can you explain LLMs?}\quad\to\quad x_t=\text{Sure}$
- **Step2:** $\mathbf{x}=\text{Can you explain LLMs? Sure}\quad\to\quad x_t=\text{.}$
- **Step3:** $\mathbf{x}=\text{Can you explain LLMs? Sure.}\quad\to\quad x_t=\text{Large}$
- **Step4:** $\mathbf{x}=\text{Can you explain LLMs? Sure. Large}\quad\to\quad x_t=\text{Language}$
- $\dots$

GPT와 같은 Decoder-only 트랜스포머는, 새로운 토큰을 생성할 때마다 지금까지의 입력과 생성된 모든 토큰을 다시 모델에 입력한다.

![fig1](AI/Transformer/KV_Cache-1.png){: style="display:block; margin:0 auto; width:100%;"}

문제는 이 과정에서 Self-Attention 연산이 매 단계마다 반복된다는 점이다.

트랜스포머의 Self-Attention은 입력으로부터 파생된 Query, Key, Value 벡터를 이용해서 다음과 같이 계산된다.

$$
\text{Attention}(Q,K,V)=\text{softmax}(\frac{QK^\top}{\sqrt{d_k}})V
$$

이때 $Q$는 현재 생성하려는 토큰 $\mathbf{x}\_{t}$가, $K$와 $V$는 이전까지의 토큰들 $\mathbf{x}\_{1:t-1}$이 생성한다.

$$
Q=W_Q\cdot\mathbf{x}_t\quad,\quad K=W_K\cdot\mathbf{x}_{1:t-1}\quad,\quad V=W_V\cdot\mathbf{x}_{1:t-1}
$$

따라서 새로운 토큰이 생성될 때마다 과거 토큰들에 대한 Key와 Value를 매번 다시 계산해야 한다.
<br>
이미 이전 단계에서 계산했던 값임에도 불구하고 동일한 연산이 반복되므로, 시퀀스 길이 $N$이 길어질수록 계산량과 추론 시간이 크게 증가하게 된다.

바로 이 비효율성을 해결하기 위해 등장한 기법이 KV Cache이다.
<br>
KV Cache는 이전 단계에서 계산한 Key와 Value를 저장해 두고 재사용함으로써, 불필요한 중복 연산을 제거하고 추론 속도를 크게 향상시킨다.

KV Cache 매 Step마다 이전 토큰들의 K와 V 벡터를 다시 계산하는 대신, 한 번 계산한 K, V 값을 GPU VRAM에 임시로 저장해 둔다. 

다음 토큰을 생성할 때는 새로 들어온 토큰의 Q, K, V만 계산한 뒤, 메모리에 저장해 둔 과거의 K, V를 불러와서 Attention 연산을 수행한다.

이 덕분에 불필요한 반복 연산이 사라져 추론 속도가 획기적으로 빨라집니다.

### KV Cache의 Memory-Bound Problem

KV Cache 덕분에 '연산량(Compute)' 문제점은 해결되었지만, 대신 '메모리(Memory)'라는 새로운 장벽을 만나게 됩니다.

KV Cache는 문장이 길어질수록, 그리고 한 번에 처리하는 사용자 요청(Batch Size)이 많아질수록 GPU VRAM을 무섭게 집어삼킵니다.

* **모델 가중치(Weights):** 크기가 고정되어 있습니다. (예: 7B 모델은 약 14GB 차지)
* **KV Cache:** Context 길이에 비례하여 선형적으로 계속 증가합니다.

결국 LLM 서버를 운영하다 보면, 연산 성능(GPU 코어)은 남아도는데 KV Cache를 저장할 VRAM이 부족해서 서버가 터지는(OOM, Out of Memory) 현상이 발생합니다. 이를 **Memory-Bound** 상태라고 부릅니다.

## 4. 최신 최적화 트렌드 (어떻게 해결할까?)

KV Cache의 거대한 용량을 줄이기 위해 AI 연구진들은 다양한 기법을 도입하고 있습니다.

* **MQA (Multi-Query Attention) / GQA (Grouped-Query Attention):** 여러 개의 Query 헤드가 소수의 Key, Value 헤드를 공유하도록 구조를 변경하여 KV Cache의 절대적인 크기 자체를 줄입니다. (Llama 등 최신 모델에 기본 적용)
* **PagedAttention:** OS의 가상 메모리 페이징 기법을 차용하여, 파편화되는 KV Cache 메모리를 블록 단위로 효율적으로 할당합니다. (vLLM 프레임워크의 핵심 기술)
* **KV Cache Quantization:** K, V 벡터를 FP16(16비트)에서 INT8이나 INT4(8/4비트)로 압축하여 화질(성능) 저하를 최소화하면서 용량을 절반 이하로 줄입니다.
