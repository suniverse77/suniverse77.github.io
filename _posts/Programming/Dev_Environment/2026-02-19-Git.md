## Git이란?

Git은 내 컴퓨터에서 코드의 모든 변경 사항을 기록하고 관리하는 버전 관리 소프트웨어이다.

### Git vs GitHub

**Git**은 내 컴퓨터에 설치하는 소프트웨어이기 때문에 로컬 저장소에서 작동하며, 다른 개발자와 작업을 공유하고 협업하기 어렵다.

**GitHub**는 온라인에 존재하는 웹 서비스로, Git으로 관리하는 프로젝트를 클라우드 저장소에 올려 다른 사람과 공유하고 안전하게 협업할 수 있는 공간을 제공한다.

즉, Git은 내 컴퓨터에서 버전 관리를 해주는 도구이고, GitHub는 그 결과물을 온라인에 올려 다른 사람들과 소통하게 해주는 저장소이다.

## Git 기능

Git에는 여러 기능이 있으며, 이를 이용하면 프로젝트의 추적과 분업 및 협업이 수월해진다.

### History

코드의 변경 사항을 시간 순으로 기록하는 기능이며, 단순히 파일을 저장하는 것이 아니라 '누가, 언제, 어떤 이유로' 코드를 수정했는지 상세히 기록한다.

이 덕분에 실수가 발생했을 때 언제든 과거의 특정 시점으로 되돌릴 수 있다.

기록을 남기는 최소 단위를 commit이라고 하며, History는 이 commit들이 모여 만들어진다.

| 명령어 | 설명 |
| :---: | :---: |
| `git add {파일명}` | 수정된 파일을 스냅샷을 찍기 위한 Staging Area에 올림 <br> (모든 수정 파일을 올리려면 `git add .`) |
| `git commit -m {설명}` | Staging Area에 있는 파일들을 하나의 스냅샷으로 기록 |
| `git log --oneline` | 지금까지의 commit 이력을 한 줄씩 간결하게 출력 |
| `git restore {파일명}` | 마지막 commit 상태로 파일을 되돌림 |


### Branch

하나의 프로젝트에서 Main Branch를 복사하여 독립적인 작업 공간을 만드는 기능이다.

메인 코드에 영향을 주지 않고 새로운 기능을 실험하거나 버그를 수정할 수 있는 일종의 평행 세계를 제공하며, 이 덕분에 하나의 프로젝트 안에서 여러 가지 실험을 독립적으로 진행할 수 있다.

| 명령어 | 설명 |
| :---: | :---: |
| `git branch {생성할 branch 이름}` | 새로운 branch 생성 |
| `git switch {branch 이름}` | 해당 branch로 작업 공간 이동 |
| `git branch -d {삭제할 branch 이름}` | 작업이 끝난 branch 삭제 |

### Merge

각기 다른 Branch에서 작업한 결과물을 다시 하나의 메인 코드로 합치는 기능이다.

| 명령어 | 설명 |
| :---: | :---: |
| `git merge {가져올 branch 이름}` | 현재 내가 위치한 branch로 <br> `{가져올 branch 이름}`의 작업을 가져와 통합 |

## Git을 통한 협업

로그인 기능을 담당하는 **개발자 A**와 장바구니 기능을 담당하는 **개발자 B**가 하나의 쇼핑몰 프로젝트를 함께 만드는 상황을 가정해 보자.
<br>
두 사람이 각각 `main` 코드에서 자신의 브랜치를 파서 작업을 시작했을 때 발생할 수 있는 상황에 대해 살펴보자.

### 서로 다른 파일을 작업하는 경우 (충돌 없음)

1. A가 `branch-A`에서 로그인 기능을 완성하고, 깃허브 `main` 브랜치에 코드를 먼저 올린다. (`main`이 업데이트됨)
2. 이후 B도 `branch-B`에서 장바구니 기능을 완성해서 깃허브에 코드를 올리려고 한다. 
3. 하지만 깃허브는 'A가 `main`을 이미 업데이트했으니, 그 코드부터 먼저 가져가`라며 B의 push를 거절한다.
4. B는 `git pull origin main`을 입력해, A가 만든 로그인 코드를 자기 브랜치로 가져온다. 
    <br>
    이때 A와 B가 작업한 폴더가 서로 겹치지 않기 때문에, Git이 알아서 두 코드를 충돌 없이 합쳐준다. 
6. 병합이 무사히 완료되면, B는 다시 안전하게 코드를 `push` 한다.

### 서로 같은 파일을 고쳤지만, 수정한 위치가 다른 경우 (충돌 없음)

1. A와 B 모두 `index.html` 파일을 열어서 작업한다.
2. A는 파일의 맨 위쪽에 로그인 버튼을 달고, 깃허브 `main` 브랜치에 코드를 먼저 올린다. (`main`이 업데이트됨)
3. 이후 B는 파일의 맨 아래쪽에 장바구니 버튼을 달고 깃허브에 코드를 올리려고 한다.
4. 하지만 깃허브는 'A가 `main`을 이미 업데이트했으니, 그 코드부터 먼저 가져가'라며 B의 push를 거절한다.
5. B는 `git pull origin main`을 입력해, A가 수정한 최신 코드를 자기 브랜치로 가져온다.
    <br>
    이때 둘 다 같은 파일을 건드렸지만 수정한 줄이 다르기 때문에, Git이 알아서 두 코드를 충돌 없이 합쳐준다.
6. 병합이 무사히 완료되면, B는 다시 안전하게 코드를 `push` 한다.

### 같은 파일의 같은 위치를 고친 경우 (충돌 발생)

1. A와 B 모두 `index.html`의 배경색 코드를 수정한다.
2. A는 배경색을 빨간색으로 수정하고, 깃허브 `main` 브랜치에 코드를 먼저 올린다. (`main`이 업데이트됨)
3. 이후 B는 자기 컴퓨터에서 배경색을 파란색으로 수정하고 깃허브에 코드를 올리려고 한다.
4. 하지만 깃허브는 'A가 `main`을 이미 업데이트했으니, 그 코드부터 먼저 가져가'라며 B의 push를 거절한다.
5. B는 `git pull origin main`을 입력해, A가 수정한 최신 코드를 자기 브랜치로 가져온다.
    <br>
    하지만 이번에는 같은 파일의 똑같은 줄을 서로 다르게 수정했기 때문에, Git이 어떤 색을 선택해야 할지 몰라 병합을 멈추고 **충돌 (Merge Conflict) 경고**를 띄운다.
6. B는 팀원과 상의하여 최종 배경색을 합의한 뒤, 충돌이 난 코드를 직접 알맞게 수정하고 저장한다.
7. 충돌을 직접 해결한 B는 다시 `git add`와 `commit`을 거친 뒤, 안전하게 코드를 `push` 한다.

### 내 작업이 안 끝났는데, main의 최신 코드가 당장 필요한 경우

1. A가 로그인 기능을 완성하여 깃허브 `main` 브랜치에 코드를 먼저 올린다. (`main`이 업데이트됨)
2. B는 `branch-B`에서 장바구니 기능을 만들던 중, A가 방금 올린 최신 로그인 코드가 당장 필요해졌다. 하지만 B의 작업은 아직 반밖에 끝나지 않았다.
3. 작업이 안 끝난 상태에서 남의 코드를 무작정 가져오면 코드가 꼬이거나 날아갈 수 있으므로, B는 현재까지 작성한 미완성 코드를 일단 `commit` 하여 컴퓨터에 안전하게 저장해둔다.
4. 이후 B는 `git pull origin main`을 입력해, A가 수정한 최신 코드를 자기 브랜치로 가져온다.
    <br>
    `commit`으로 코드를 저장했기 때문에, 만약 Git이 병합할 때 충돌하는 부분이 있다면 에러를 발생시키지 않고 사용자에게 물어보면서 병합하게 된다.
5. 최신 코드를 무사히 가져오면, B는 다시 작업을 계속 이어간다.
